# Architecture Perspective Checklist

アーキテクチャ観点でのレビューチェックリスト。
プロジェクトのアーキテクチャパターン（レイヤード / Hexagonal / Clean Architecture 等）に基づいて適用する。

## Must Fix

### レイヤー違反・依存逆転
- [ ] domain 層が infrastructure / presentation 層に依存していないか
- [ ] repository 実装が domain 層のインターフェースを経由しているか
- [ ] presentation 層（API / Controller）が直接 DB アクセスしていないか（service 層を経由すべき）
- [ ] DI 設定モジュールのみが具象実装を参照しているか

### モジュール境界違反
- [ ] 他モジュールの内部クラス（`internal` / `impl` パッケージ）を直接参照していないか
- [ ] モジュール間の循環依存が発生していないか
- [ ] モジュール間の依存が公開インターフェース経由になっているか

### DI 違反
- [ ] `new` で具象実装を直接生成していないか（DI コンテナ経由が原則）
- [ ] DI アノテーション / 設定の使い方がプロジェクト規約に沿っているか

## Should Fix

### 責務の越境
- [ ] Service クラスに Repository の責務（クエリ組み立てなど）が混入していないか
- [ ] Controller / Resource クラスにビジネスロジックが記述されていないか
- [ ] DTO 変換ロジックが適切な場所にあるか

### 設計パターン不整合
- [ ] Entity / ValueObject の区別が適切か（ID の有無、不変性）
- [ ] 不変オブジェクトの生成パターンが既存コードと一致しているか
- [ ] ORM Entity の定義がテーブル設計と一致しているか

### API 設計
- [ ] RESTful 規約に従っているか（リソース命名、HTTP メソッド選択）
- [ ] エンドポイントの粒度が適切か（過度に細かい / 粗い）
- [ ] レスポンス構造が既存 API と一貫しているか

## Suggestion

### パッケージ / ディレクトリ構成
- [ ] パッケージ / ディレクトリ命名が既存規約と一致しているか
- [ ] ファイル配置が該当モジュールの標準構成に従っているか

### 共通化の余地
- [ ] 類似処理が複数箇所にある場合、共通化を検討しているか
- [ ] ユーティリティクラスの再利用を検討しているか

### ドキュメントとの整合性
- [ ] API 設計書と実装が乖離していないか
- [ ] DB 設計書と Entity 定義が一致しているか
