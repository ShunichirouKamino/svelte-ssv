# Maintainability Perspective Checklist

保守性観点でのレビューチェックリスト。
コードの可読性、一貫性、拡張性に焦点を当てる。

## Should Fix

### 命名
- [ ] クラス名・メソッド名・変数名が処理内容を正確に表しているか
- [ ] 既存コードの命名規約と一致しているか
- [ ] 略語の使用が最小限か（ドメイン固有の略語を除く）
- [ ] boolean 変数 / メソッドは `is` / `has` / `can` / `should` で始まっているか

### コード重複
- [ ] 同一ロジックが複数箇所にコピーされていないか
- [ ] 共通化可能な処理が適切に抽出されているか
- [ ] テストコードの重複も含めて確認（テストユーティリティの活用）

### 複雑度
- [ ] メソッド / 関数の行数が過度に長くないか（目安: 50 行以下）
- [ ] ネストの深さが 3 段以上になっていないか
- [ ] 条件分岐が複雑な場合、ガード句やポリモーフィズムで整理できないか

### 型安全性
- [ ] 型の不整合がないか
- [ ] 不適切な型変換（キャスト）がないか
- [ ] 型パラメータの raw type 使用がないか（該当言語の場合）

## Suggestion

### ドキュメンテーション
- [ ] public クラス・メソッドにドキュメントコメントが付与されているか
- [ ] パラメータ・戻り値・例外の説明が含まれているか
- [ ] ビジネスロジックの意図が説明されているか

### コードスタイル
- [ ] プロジェクトのフォーマッタ / リンター規約に準拠しているか
- [ ] import / require 文が整理されているか（未使用、ワイルドカードの排除）
- [ ] マジックナンバー / マジックストリングが定数化されているか

### テスト
- [ ] テストメソッド名がテスト内容を説明しているか（プロジェクトの命名規約に従う）
- [ ] テストの Arrange-Act-Assert パターンが明確か
- [ ] エッジケースのテストが含まれているか

### 拡張性
- [ ] 変更が閉じた範囲に収まっているか（OCP: Open-Closed Principle）
- [ ] 設定値がハードコードされていないか（外部設定化を検討）
- [ ] 将来の変更点が予測可能な設計になっているか
