# Documentation Perspective Checklist

ドキュメント（`.md` ファイル）観点でのレビューチェックリスト。
冗長化・二重管理の排除に焦点を当てる。

対象: PR の差分に `.md` ファイルが含まれる場合に適用する。

## Must Fix

### 二重管理の検出

- [ ] 設定ファイル（`.env`, `settings.json`, `.claudeignore`, `tsconfig.json` 等）の具体的な値をドキュメントに転記していないか
  - Bad: ドキュメントに JSON の全文をコピペ → 設定変更時にドキュメントも更新が必要
  - Good: 「具体的な設定値は [`.claude/settings.json`](パス) を参照」+ 方針・意図のみ記載
- [ ] コードの実装詳細（関数シグネチャ、クラス構造等）をドキュメントに転記していないか
  - Bad: ドキュメントに型定義やコード全文をコピペ
  - Good: 「実装の詳細は `server/agent.ts` を参照」+ 設計意図のみ記載
- [ ] 同一内容が複数のドキュメントに記載されていないか
  - 同じ手順・設定が複数ファイルに書かれている場合は、1箇所に集約して他からはリンク参照

### 参照先の妥当性

- [ ] 参照先として示されたファイルが実在するか
- [ ] 参照先のリンクパスが正しいか（相対パスの場合、ドキュメントの配置場所から正しく解決されるか）

## Should Fix

### 方針・意図の記載

- [ ] 設定値やコードを参照する場合、「なぜその設定にしているか」の方針が記載されているか
  - Bad: 「`.env` は読み取り禁止」（what だけ）
  - Good: 「`.env` は API キー・DB 接続情報を含むため、AI エージェントによる読み取りを禁止」（why を含む）
- [ ] 除外・禁止のルールに対して、理由が明記されているか
- [ ] 表形式（対象 / 理由）で整理されているか（箇条書きの羅列より表の方が一覧性が高い）

### 内容の重複・冗長

- [ ] 同じ情報が同一ドキュメント内で繰り返されていないか
- [ ] 箇条書きの項目が冗長でないか（1項目1行を目安に簡潔にする）
- [ ] 「はじめに」「概要」等の前置きが長すぎないか（読者はすぐに本題に入りたい）

### 構造

- [ ] 見出しの階層が適切か（H2 → H3 → H4 の順。H1 はドキュメントタイトルのみ）
- [ ] 目次が必要な長さ（50行超）のドキュメントに目次がないか
- [ ] セクション番号が連番になっているか（欠番や重複がないか）

## Suggestion

### 保守性

- [ ] ドキュメント更新時に他のドキュメントの更新が必要になる構造になっていないか
  - 理想: 1つの事実は1箇所にだけ書かれている（DRY 原則のドキュメント版）
- [ ] 将来の変更に対して脆い記載がないか（バージョン番号のハードコーディング等）
- [ ] 「現時点では」「暫定的に」等の期限付き記載に、見直し時期が明記されているか

### 読みやすさ

- [ ] 対象読者に合った表現か（エンジニア向け / 非エンジニア向け）
- [ ] 略語・専門用語の初出時に説明があるか
- [ ] コマンド例やコード例がコピペで実行可能か（パスやプレースホルダが明示されているか）
