# Reliability Perspective Checklist

信頼性観点でのレビューチェックリスト。
例外処理、トランザクション管理、データ整合性に焦点を当てる。

## Must Fix

### 例外処理
- [ ] 例外の握りつぶし（空の catch ブロック）がないか
- [ ] `catch (Exception e)` で広すぎる例外をキャッチしていないか
- [ ] ビジネス例外とシステム例外を適切に区別しているか
- [ ] 例外発生時にリソースが解放されるか（finally / try-with-resources）

### トランザクション境界
- [ ] トランザクション境界が適切か（Service 層で管理されているか）
- [ ] 複数テーブルへの更新が同一トランザクション内で行われているか
- [ ] 読み取り専用操作に不要なトランザクションが付与されていないか
- [ ] ネストされたトランザクションの伝播設定が正しいか

### データ整合性
- [ ] 一意制約違反が適切にハンドリングされているか
- [ ] 楽観ロック（`@Version`）が必要な箇所で使用されているか
- [ ] Nullable / non-nullable handling is consistent between schema definition and application code

## Should Fix

### 境界条件
- [ ] null チェックが必要な箇所で行われているか
- [ ] コレクションの空チェックが行われているか
- [ ] 数値のオーバーフロー / アンダーフローを考慮しているか
- [ ] 日付の境界（月末、年末、閏年）を考慮しているか

### 外部連携
- [ ] 外部 API 呼び出しのタイムアウトが設定されているか
- [ ] 外部サービス障害時のフォールバック / サーキットブレーカーがあるか
- [ ] 冪等性が必要な操作で担保されているか

### ステータスコード
- [ ] HTTP ステータスコードが適切か（200/201/204/400/404/409/500 等）
- [ ] エラーレスポンスの形式が統一されているか

### ログ
- [ ] 例外発生時に十分な情報がログ出力されているか
- [ ] 重要な業務操作の開始・完了がログに記録されているか
- [ ] ログレベルが適切か（ERROR / WARN / INFO / DEBUG）

## Suggestion

### リトライ
- [ ] 一時的な障害に対するリトライ機構を検討しているか
- [ ] リトライ時のバックオフ戦略が設定されているか
- [ ] リトライ上限が設定されているか

### ヘルスチェック
- [ ] 新しい外部依存を追加した場合、ヘルスチェックに含まれているか

### テスタビリティ
- [ ] 例外パスのテストが可能な設計になっているか
- [ ] 外部依存がモック可能な構造になっているか
